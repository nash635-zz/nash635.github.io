ValidationService@WASP

Motivation

We have several different requirements for validation and scroing, with a standard service, we can have users focus with their business logic and handel below items for them.

  • One optimization, opt-in everywhere.
  • Reduce manual work.
  • Store validation results in DB and easy to check the history.

F.A.Qs

How can I monitor the backend experiment status?

[Wasp]: Check the jobid from stdout of service module, and go to Web Portal to see the record.

Is the metrics long term saved?

[Wasp]: Yes, we save them to database and will never be expired, you can retrive it with your jobid with the link provided by web dashboard. The result can be retrived with the JobId like this link.

What will be supportted next?

[Wasp]: Structured input for validation service and trainer, user can define a pipeline for each step(training,validation…) through json format input.

Is there a limitation for the total slice combination count?

[Wasp]: Yes, we control it not larger than 1000, as if a user set some kind of id column as slice by mistake, our DB will be inserted a unexpected large number of records.

Why User need to provid the input in binary format, and how to debug?

[Wasp]: Binary data is more efficient. It works well in normal case. For debugging, we have tool to convert it into text, like the feature vector, and model format, please contact @fesun/@xianzh for latest version and how to use it.
